# FlashLingo `.flashjp` package format

This document summarizes the package contract consumed by
`lib/features/importer/importer_service.dart`. For the full `manifest.json`
reference, see [manifest-json-mazos.md](manifest-json-mazos.md).

## Accepted Extensions

FlashLingo accepts package files with this extension:

- `.flashjp`

Plain `.zip` files are not accepted by the app. A ZIP renamed to `.flashjp` is
accepted only when it is an app-generated user backup with a valid
`backup_manifest.json` using `flashlingo_user_backup_v1`. Official deck
packages remain signed/encrypted containers.

Accepted `.flashjp` profiles:

- official deck package: binary container that starts with `FLJPENC1\n`,
  includes a signed JSON header, and stores the original ZIP package as an
  AES-256-GCM encrypted payload.
- user progress backup: ordinary ZIP generated by the app, with
  `backup_manifest.json.package_kind = user_backup` and
  `backup_manifest.json.format = flashlingo_user_backup_v1`.

## Official Deck Packages

Accepted encrypted formats:

- `flashlingo_encrypted_v2`: legacy large-deck format. It signs only the
  protected header; the header contains the SHA-256 of the ciphertext and the
  SHA-256 of the decrypted ZIP.
- `flashlingo_encrypted_v3`: current generated format. It uses the same signed
  protected header model, but encrypts the ZIP payload as independent 4 MiB
  AES-256-GCM chunks so Android can decrypt large decks with native crypto
  without loading the full payload into memory.

The importer verifies:

- file extension is `.flashjp`
- container magic is `FLJPENC1\n`
- publisher is `FlashLingo - Javier Ponce`
- publisher role is `official`
- signing key id matches the bundled Ed25519 public key
- encryption key id matches the bundled AES key
- ciphertext SHA-256 matches the header
- Ed25519 header signature is valid
- decrypted ZIP SHA-256 matches the header

Security model:

- The Ed25519 signature is the authenticity/integrity boundary: only packages
  signed with the private publisher key should import as official packages.
- The AES key used to decrypt official packages is bundled in the Flutter app so
  the app can import decks offline. Treat encryption as package container
  protection, not as DRM or a confidentiality guarantee against reverse
  engineering of the client.
- The Ed25519 private signing key must never be shipped with the app or stored
  in the repository.

Only after those checks does the importer decrypt the ZIP to a temporary app
file. It then reads the ZIP central directory, reads `manifest.json`, writes
only the SQLite database to temporary storage, copies referenced media entries
directly into the app-local media directory, and deletes the temporary ZIP. It
does not decode or extract the full ZIP archive just to inspect package
metadata.

## Official Deck Archive Layouts

The decrypted ZIP payload accepts both layouts:

1. Flat archive contents.
2. A single wrapper directory containing the package contents.

Examples:

```text
manifest.json
flashcards.db
audio/word_001.mp3
images/card_001.png
```

```text
starter_pack/
  manifest.json
  flashcards.db
  audio/word_001.mp3
  images/card_001.png
```

If more than one `manifest.json` exists, the importer uses the one whose path is
closest to the archive root.

## Official Deck SQLite Contract

The importer reads a table named `flashcards`.

Columns currently consumed:

- `ID`
- `PALABRA`
- `SIGNIFICADO`
- `AUDIO_PALABRA`
- `AUDIO_ORACION`
- `IMAGEN`
- `ORACION`
- `TRADUCCION`
- `LECTURA_PALABRA`
- `LECTURA_ORACION`
- `FORMAS`

Each SQLite row produces two FlashLingo cards:

- recognition card: `{language_id}_recog`
- production card: `{language_id}_prod`

The logical identity used during import/update is:

```text
originalId + cardType
```

In code this is stored as:

```text
{originalId}::{cardType}
```

## Official Deck Card Mapping

Recognition card:

- question: `PALABRA`
- answer: `SIGNIFICADO`
- sentence: `ORACION`
- translation: `TRADUCCION`
- extra data: `LECTURA_PALABRA`, `LECTURA_ORACION`, `FORMAS`, type
  `recognition`

Production card:

- question: `SIGNIFICADO`
- answer: `PALABRA`
- sentence: `TRADUCCION`
- translation: `ORACION`
- extra data: target reading, sentence reading, `FORMAS`, type `production`

Both card types reuse word audio, sentence audio, image, and initial SRS decay
rate from the imported package/settings.

## Official Deck Media Resolution

Referenced media files are copied into the app-local `media_assets` directory
using UUID file names. The importer skips `.db` files, `manifest.json`, files
without an extension, and media entries that are not referenced by SQLite rows
or the deck icon field.

Lookup is attempted by:

1. exact path relative to the selected package root
2. exact path inside the internal ZIP
3. basename match only when the SQLite/manifest reference is also a basename
4. decoded and lowercase variants of those exact rules
5. extensionless stem match within the same media kind

The stem fallback allows `AUDIO_PALABRA = de-DE00001_PALABRA.wav` to resolve to
`media/de-DE00001_PALABRA.mp3`, but only because both sides are audio. An image
with the same stem is not copied for an audio reference, and an audio file is
not copied for an image/icon reference. Files without an extension are skipped
as media. ZIP creators should store
already-compressed audio/image/video formats such as `.mp3`, `.jpg`, `.png`,
`.webp`, `.ogg`, `.opus`, and `.mp4` with ZIP method `STORE` instead of
`DEFLATE`.

## User Progress Backups

FlashLingo can export one local deck as a `.flashjp` user backup and later
import it through the same file picker. This backup profile is intentionally
separate from official signed deck packages:

- It is generated on the device by `DeckProgressBackupService`.
- It uses extension `.flashjp` for user convenience.
- It is a ZIP file, not an `FLJPENC1` encrypted official container.
- It does not contain the official signing private key and cannot create
  official `flashlingo_encrypted_v3` packages.
- v1 has no user password and no extra encryption. Anyone who receives the file
  can inspect/import its contents.

Required root files:

- `backup_manifest.json`
- `cards.json`
- `settings.json`
- `progress.json`
- optional `media/...` files referenced by cards or the deck icon

`backup_manifest.json` declares:

- `package_kind = user_backup`
- `format = flashlingo_user_backup_v1`
- `format_version = 1`
- app/version metadata
- export timestamp
- source deck name and ISO code
- counts for cards, review logs, session history, daily stats, and media files
- SHA-256 and size for every declared JSON/media file

`cards.json` contains full card content plus the portable card key:

```text
originalId::cardType
```

Each card includes question, answer, sentence, translation, extra JSON,
restorable media references, SRS state, next/last review dates, learning queue,
decay rate, lapse counters, lifetime review counters, and total study time.

`settings.json` contains all `DeckSettings` fields, including daily counters,
reminders, SRS parameters, learning steps, write mode, undo, study mix mode, and
deck icon reference.

`progress.json` contains:

- `review_logs`, remapped on restore from portable card keys to new local
  `Flashcard.id` values
- `active_session`, including `queue_card_keys` that are remapped to local IDs
- `session_history`
- `daily_stats`

Restore behavior:

- If the source deck name does not exist, the backup imports as a new deck.
- If the source deck name exists, the UI always asks whether to restore over
  the existing deck or create a copy with another name.
- Restore-over-existing deletes that deck's current cards, settings, logs,
  active sessions, history, and daily stats, then writes the backup data.
- Create-copy writes the same content/progress under the chosen new deck name.
- Media is copied into the app-local `media_assets` directory with new UUID
  filenames.
- `DeckDailyStats` rows are restored and then synchronized for affected study
  days from logs/session history so derived values stay consistent.

User backup safety limits:

- max ZIP file size: `1024 MiB`
- max real file entries: `12000`
- max single entry: `128 MiB`
- max single media file: `64 MiB`
- max total media bytes: `768 MiB`
- only relative safe ZIP paths are accepted; absolute paths, `..`, `.`, empty
  segments, and path segments containing `:` are rejected
- declared hashes and sizes must match actual file contents
- manifest counts must match JSON contents
- duplicate card keys and progress references to missing cards are rejected

## Importer Safety Limits

For official deck packages, the importer decrypts a verified `.flashjp`
container, reads the internal ZIP central directory with safe path validation,
and rejects suspicious ZIP paths. It also applies size limits:

- max real file entries: `12000`
- max single extracted file: `128 MiB`
- max total extracted bytes: `1024 MiB`
- max single media file: `64 MiB`
- max total media bytes: `768 MiB`

SQLite rows are read in batches of `250`; Isar Community card writes are flushed in
batches of `400`.

## Current Importer Guarantees

- Plain ZIP files are rejected before extraction.
- A ZIP renamed to `.flashjp` is accepted only if it is a valid
  `flashlingo_user_backup_v1` backup.
- Encrypted `.flashjp` containers must pass header signature, key id,
  ciphertext hash, and plaintext hash validation before package contents are
  imported.
- User backups must pass manifest version, path, size, SHA-256, count, duplicate
  card key, and progress-reference validation before restore.
- v3 containers are decrypted in chunks; v2 containers remain accepted for
  compatibility.
- UI imports stage the selected `.flashjp` into app cache from the system file
  picker before preview/import, so the importer reads an app-owned file instead
  of a shared-storage path.
- The Riverpod importer service and controller are keep-alive providers, so a
  long preview/decryption operation can finish even when started from a one-shot
  `ref.read` flow, and the prepared preview can be reused by the import step.
- ZIP metadata is read from the central directory, and only manifest, SQLite,
  and referenced media entries are streamed from the ZIP.
- SQLite rows are read in batches.
- Isar Community writes are flushed in batches.
- Debug builds log per-phase import timings for verification/decryption, ZIP
  opening, manifest/database extraction, media reference collection, media
  copy, database reads, backup restore, and Isar Community writes.
- Orphaned media is cleaned after import/update/delete.
- The staged `.flashjp` copy is deleted when the UI import flow finishes,
  fails, or is cancelled.
- Duplicate logical cards inside the same import are ignored after the first
  matching row.

## Things The Importer Does Not Do

- It does not delete existing cards that are missing from the new package.
- It does not support remote downloads.
- It does not edit cards manually inside the app.
- It does not import progress counters such as `newCardsSeenToday` or
  `lastNewCardStudyDate` from official deck `manifest.json`. User backups do
  restore those deck settings because they are explicit user backup data.
